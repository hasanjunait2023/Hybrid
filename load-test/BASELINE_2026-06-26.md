# k6 Storefront Load-Test Baseline — 2026-06-26

## Test parameters
- Script: `load-test/storefront-load.js`
- Target: `https://hybrid.ecomex.cloud` (apex marketing + tenant subdomain rewrites)
- Tool: k6 v0.50.0 (`/usr/local/bin/k6`)
- Mode: ramping-vus, 0 → 5 → 5 (30s hold) → 0
- Virtual users: 5
- Duration: 30s steady state (45s total including ramp)
- Iterations: 35–36 complete
- Total requests: 70

## Interpretation

- **5 VU is very light load.** ~1.5 req/s aggregate. This is well below the
  scale triggers from `docs/SCALING_PLAN.md` (Phase B at hundreds of tenants,
  Phase C at thousands).
- **p95 of 600 ms for the storefront page** is on the slow end of the SLO
  target (<150 ms edge-cached, <250 ms origin). The current apex serves HTML
  with `cf-cache-status: DYNAMIC` because Next.js renders every request (multi-
  tenant rewrite path; intentional per `_PHASE_P1_PERF_REPORT.md`).
- **group p95 = 2.6 s** includes marketing signup render which hits
  more DB paths than the home page.
- **0% failures** under 5 VU — well within budget.

## Next baseline targets

- After S1 (CF wildcard TLS + edge cache): aim for storefront p95 < 200 ms
  edge-cached (50 ms on hit), group p95 < 1.5 s.
- Stress test at VUS=50, 100: find the inflection point where p95 crosses 1s.
  That signals the single-VPS ceiling (Phase B trigger).
