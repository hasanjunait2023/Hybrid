# S1.H6 Fix — supabase-meta healthcheck upstream bug

## Problem

`supabase-meta-pe9o2li2n3bns3wnofob49uw` showed "Up 38 hours (unhealthy)" since
2026-06-25. Investigation:

- **Healthcheck (from `supabase/postgres-meta:v0.95.2` image)**:
  `node -e "fetch('http://localhost:8080/health').then((r) => {if (r.status !== 200) throw new Error(r.status)})"`
- Healthcheck config: `interval=5s timeout=5s retries=3`
- Manual probe: `/health` returns HTTP 200, but **cold-start is 1896ms** —
  within the 5s budget only by luck. Subsequent calls are 200-300ms (warm).
- FailingStreak grew to 11; restart brought it to 4 then back up.
- Compose file (`/data/coolify/services/pe9o2li2n3bns3wnofob49uw/docker-compose.trimmed.yml`)
  had no `healthcheck:` block for `supabase-meta` — the healthcheck was
  baked into the upstream image.

## Fix (2026-06-27)

Added an explicit `healthcheck:` override to the compose file for the
`supabase-meta` service. Bumped all timeouts to give the `/health` query
enough headroom:

```yaml
healthcheck:
  test:
    - CMD-SHELL
    - 'node -e "fetch(''http://localhost:8080/health'').then((r) => { if (r.status !== 200) throw new Error(r.status); })"'
  interval: 30s
  timeout: 30s
  retries: 5
  start_period: 30s
```

**Why these values:**
- `interval: 30s` — less log spam, matches DB query cadence
- `timeout: 30s` — covers the cold-start of ~2s with huge margin
- `retries: 5` — at 30s interval, gives 2.5 min of grace before failing
- `start_period: 30s` — meta is one of the last services to come up;
  need warm-up time before judging health
