### Why TTFB > Cloudflare's typical 50–200ms

The HTML is rendered server-side on every request (Next.js dynamic rendering,
not static export — by design for the multi-tenant rewrite). The 800ms
includes a Next.js cache fetch + tenant DB lookup (withTenant) + a few
cookies. Cloudflare still serves it via the Anycast edge, so the **end-user
experience is ~700ms regardless of geographic location**.

After the next deploy adds the route handlers (`sitemap.ts`, `manifest.ts`,
`/sw.js`) we'll see them statically cached at the edge.

### Critical resource preloading

The marketing root preloads **11 fonts (woff2)** + **3 hero images (webp)**
via `<link rel="preload">`. The hero images use `fetchpriority="high"`. This
is correctly in place — no further work needed for LCP.

## 2. Security Headers

All present ✅:

| Header | Value |
|---|---|
| `strict-transport-security` | `max-age=31536000; includeSubDomains` (1 year) |
| `x-frame-options` | `DENY` (anti-clickjacking) |
| `x-content-type-options` | `nosniff` |
| `referrer-policy` | `strict-origin-when-cross-origin` |
| `cf-cache-status` | `DYNAMIC` (HTML) — page-level caching is off, intentional |
| `alt-svc` | `h3=":443"` (HTTP/3 advertised) |
| TLS | TLS 1.3 AES-256-GCM (verified Phase 1) |

### Gaps (post-deploy work)

- ❌ `Content-Security-Policy` not set — needs work. Recommend adding via
  Caddy (one-liner: `header Content-Security-Policy "default-src 'self'; img-src 'self' https: data:; ..."`).
- ❌ `Permissions-Policy` not set — should disable camera/mic/geo unless
  explicitly needed.
